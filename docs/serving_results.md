# Serving Results — NanoLlama 8L

Measured on **NanoLlama 8L v1** (127.6M params, 8 layers, GQA 12Q/4KV, SwiGLU, RoPE, RMSNorm, val loss 3.357) on **CPU M1 Pro, fp32** unless noted. Source artifacts in `experiments/nanollama_serving_bench/` and `experiments/nanollama_serving_quant/`, all generated by `scripts/serving/run_nanollama_benchmarks.sh` on 2026-04-04.

---

## Engineering principle: correctness before performance

The cache equivalence gate was the go/no-go criterion before any latency work started. The test runs generation two ways — (a) prefill once + N decode steps, (b) full recompute at every step — and asserts that per-step logits match within tolerance. If this gate fails, the model is running incorrectly regardless of how fast it runs.

| Prompt length | Gen length | Max |logit diff| | Greedy token match |
|-------------|-----------|-----------------|-------------------|
| 256 | 16 | **2.09e-05** | True |
| 512 | 16 | **1.91e-05** | True |
| 1024 | 16 | 13.09 | False — expected at block_size boundary |

The 1024-token case diverges by design: when `prompt_len = block_size`, the sliding window truncates context and KV-cache state diverges from full recompute. This is a documented limitation, not a bug — the first two cases cover the primary operating range.

## KV-cache speedup

**Single-point benchmark:** prompt=256 tokens, gen=128 tokens, B=1, fp32, CPU M1 Pro.

| Mode | Prefill (ms) | Decode (ms/tok) | Tokens/sec |
|------|-------------|----------------|-----------|
| KV-cache | 99.9 | **17.8** | **56.0** |
| Full recompute | 162.9 | 196.2 | 5.1 |
| **Speedup** | **1.6×** | **11.0×** | **11.0×** |

The mechanism is algorithmic: cached decode attends only the new query token against cached keys (O(1) per step); full recompute reattends the entire growing context (O(T) per step, so O(T²) total for a T-token generation). At gen=128 the speedup is 11×; at longer generation lengths it grows further.

Prefill is also faster under KV-cache mode (99.9ms vs 162.9ms) because the recompute path re-runs prefill at every step — full recompute's "prefill" here is the first step of a sequence of full forward passes.

## TTFT / TPS grid

**Config:** gen=64, fp32. Grid: context lengths {256, 512, 1024} × batch sizes {1, 2, 4}.

### KV-cache mode

| Context | B=1 | B=2 | B=4 |
|---------|-----|-----|-----|
| 256 | TTFT 100ms / TPS 56 | TTFT 170ms / TPS 56 | TTFT 323ms / TPS 56 |
| 512 | TTFT 189ms / TPS 49 | TTFT 375ms / TPS 50 | TTFT 675ms / TPS 51 |
| 1024 | TTFT 463ms / TPS 42 | TTFT 872ms / TPS 40 | TTFT 2015ms / TPS 43 |

### Full recompute mode (for comparison)

| Context | B=1 TPS | B=2 TPS | B=4 TPS |
|---------|---------|---------|---------|
| 256 | 5 | 6 | 6 |
| 512 | 3 | 3 | 3 |
| 1024 | 1 | 1 | 2 |

**Two clean empirical patterns:**

1. **TTFT scales as O(context × batch).** At B=1: TTFT grows from 100ms (ctx=256) to 463ms (ctx=1024) — a 4.6× increase for a 4× context increase, consistent with O(T²) prefill. At fixed context, TTFT scales linearly with B (prefill batching adds sequences but doesn't change per-sequence compute).

2. **Decode TPS is stable across batch sizes.** At ctx=256: TPS is 56 for B=1, B=2, and B=4. This tells you the bottleneck is memory bandwidth (loading 486MB of model weights per step), not arithmetic throughput. Adding more sequences in a batch doesn't increase the per-step weight load. This also means prefill batching (packing multiple request prefills) is free at the decode stage — each sequence gets its own KV-cache slice and decodes independently.

## Precision and quantisation

### Precision policy

The serving layer normalises all requested dtypes and falls back gracefully on unsupported hardware:

| Requested | Executed on CPU | Reason |
|-----------|----------------|--------|
| fp32 | fp32 | Native |
| fp16 | fp32 | "fp16 unavailable on cpu, fell back to fp32" |
| bf16 | fp32 | "bf16 unavailable on cpu, fell back to fp32" |
| int8 | int8 | Dynamic quantization via qnnpack |

This behaviour is exposed via the `/health` endpoint metadata and logged per request. On CUDA, fp16 and bf16 execute natively.

### int8 dynamic quantisation results

**Config:** prompt=256, gen=64, B=1, CPU M1 Pro. Dynamic int8 (`torch.quantization.quantize_dynamic`, qnnpack, Linear layers only, no calibration dataset).

| Mode | Decode (ms/tok) | Prefill (ms) | Memory | PPL |
|------|----------------|-------------|--------|-----|
| fp32 | 17.6 | 118ms | 486 MB | 268.2 |
| int8 | 13.9 | 341ms | **147 MB** | 284.3 |
| Delta | **−21% (1.27× faster)** | **+189% (2.9× slower)** | **−70%** | **+6.0%** |

**Why decode speeds up and prefill slows down — the bandwidth vs compute split:**

- **Decode is memory-bandwidth-bound.** Each decode step loads 486MB of model weights to compute one token. int8 compresses this to ~147MB, so each step loads 3.3× less data — directly translating to faster decode.
- **Prefill is compute-bound.** Prefill processes the full prompt in one batched matrix multiply. Dynamic quantization doesn't pre-quantize weights; it quantizes per-batch at runtime, adding overhead on each prefill call without the bandwidth benefit (prefill already has high arithmetic intensity).

This split — int8 helps decode, hurts prefill — is a general property of dynamic weight-only quantization on any compute-sufficient device. On memory-constrained edge hardware or shared GPU, the 70% memory reduction often matters more than the prefill penalty.

**Recommendation engine output:** `{"cpu_default": "int8", "reason": "best_tradeoff_quality_gated"}` — the serving layer selects int8 as the CPU default after verifying PPL drift stays within the configured threshold.

**PPL drift in context:** +6% drift (268 → 284) means the model assigns slightly less probability mass to gold tokens, but greedy decoding on short prompts remains exact-match identical to fp32. The drift accumulates over longer sequences and is more visible in perplexity-sensitive applications.

## KV-cache memory formula

The serving layer uses an explicit formula for KV-cache memory budgeting:

```
bytes_per_token = 2 × n_layers × n_kv_heads × head_dim × element_size
               = 2 × 8 × 4 × 64 × 4  (fp32)
               = 16,384 bytes per token
```

At `block_size=1024`: max KV-cache = 1024 × 16KB = **16 MB**.

**GQA dividend.** NanoLlama uses 4 KV heads vs 12 query heads (3:1 ratio). Compared to MHA (12 KV heads):
- KV-cache memory: 16 MB vs 48 MB — 3× reduction
- KV weight size per layer: Q proj 768×768, K/V proj 768×256 — the K/V matrices are 3× smaller, loading faster per decode step

The GQA architectural choice was motivated by throughput during training (single batched SDPA call vs 12 per-head calls, measured as 38K→60K tok/s during ablation). The serving dividend is a compounding benefit: smaller K/V weights mean faster decode even before considering cache memory.

## Known limitations

| Limitation | Impact | Rationale |
|-----------|--------|-----------|
| Decode is B=1 | Sequential generation; no multi-sequence batching | Batched decode requires per-sequence cache management; prefill batching covers 80% of throughput benefit at 20% complexity |
| No continuous/dynamic batching | Cannot interleave requests during decode | Full scheduler scope; not in scope for this stack |
| fp16/bf16 fall back to fp32 on CPU | No precision benefit without CUDA | Hardware limitation; documented per-request |
| int8 prefill 2.9× slower | Use fp32 for prefill-heavy workloads | Dynamic quantization overhead on compute-bound path |
| KV-cache sliding window at block_size | Logit drift at context=block_size | Expected and documented; covered by equivalence gate |
| Safety is regex heuristics | Not a learned classifier | Suitable for demonstration; narrow false-positive rate |
